Spring Ioc

什么是ioc

先看官网是怎么说的：官网链接
IoC控制反转，也被称为依赖性注入（DI）。它是一个过程，对象仅通过构造方法的参数、工厂方法的参数或在对象实例被构造或从工厂方法返回后在其上设置的属性来定义其依赖关系（即它们与之合作的其他对象）。然后，容器在创建 bean 时注入这些依赖关系。这个过程从根本上说是Bean本身通过使用直接构建类或诸如服务定位模式的机制来控制其依赖关系的实例化或位置的逆过程（因此被称为控制反转）。

好拗口，我对这块的理解：没使用spring之前都是我们在类内部主动创建依赖对象，比如说汽车Car.class 需要依赖引擎Engine.class，那么我们在构造Car之时候必须先实例化一个Engine然后再传递给他，这导致类与类之间高耦合。有了 IOC 容器后，把创建和查找依赖对象的控制权交给了容器，由容器进行注入组合对象，所以对象与对象之间是松散耦合，便于测试和功能复用，整个体系结构更加灵活。

“控制反转，依赖注入”怎么说呢？

  • 控制：之前是对象控制对象，现在是容器控制对象Car，以及依赖对象Engine的获取(Car 要依赖 Engine)
  • 反转：之前是对象自己创建依赖的对象然后组装，即正转，反转则是由容器来帮忙创建及注入依赖对象。因为由容器帮我们查找及注入依赖对象，对象只是被动的接受依赖对象，所以称之为反。反转指的是依赖对象的获取被反转了。
  • 依赖：对象内部依赖的其他对象属性
  • 注入：之前是需要engine，自己实例化一个，然后赋值给Car；现在是ioc先构造一个Car(还没有engine)，从容器中获取一个engine(如果没有，就创建一个)，把engine赋值给Car

源码分析

AbstractApplicationContext#refresh

这个方法主要做spring容器配置以及bean初始化相关

AbstractApplicationContext#prepareRefresh

准备刷新容器上下文
*设置启动时间、关闭状态为false、激活状态为true，便于执行任意属性来源的初始化
*初始化上下文环境中占位符属性来演
*获取当前系统环境信息并校验
*准备早期监听器和事件集合对象，默认为空集合

AbstractApplicationContext#obtainFreshBeanFactory

刷新并获取刷新后的BeanFactory,将配置文件解析为BeanDefinition对象，加载到beanFactory中

AbstractApplicationContext#prepareBeanFactory

准备BeanFactory，设置工厂的各种属性
*设置Bean类加载器(ClassLoader)
*设置SPEL表达式解析器(可选)
*设置属性编辑器注册器
*设置Bean的后置处理器
*设置忽略的依赖接口
*注册可解析的依赖
*注册用来检测内部Bean的早期引用后置处理器
*注册默认的环境相关的bean，都是单例的

AbstractApplicationContext#postProcessBeanFactory

BeanFactory的后置处理器，该方法为空方法，具体由子类实现，使用的设计模式为模板方法中钩子方法
Spring 开放接口留给子类去实现该接口，主要用来改变 BeanFactory。

不过GenericWebApplicationContext重写了该方法 主要做了如下事情
*增加bean的后置处理器
*增加要忽略的依赖接口
*注册web应用的scope
*注册web环境相关的bean

AbstractApplicationContext#invokeBeanFactoryPostProcessors

实例化并调用所有注册的beanFactory后置处理器，BeanFactoryPostProcessor 接口是 Spring 初始化 BeanFactory 时对外暴露的扩展点，Spring IoC 容器允许 BeanFactoryPostProcessor 在容器实例化任何 bean 之前读取 bean 的定义，并可以修改它。

本方法会实例化和调用所有 BeanFactoryPostProcessor（包括其子类 BeanDefinitionRegistryPostProcessor）。BeanDefinitionRegistryPostProcessor 继承自 BeanFactoryPostProcessor，比 BeanFactoryPostProcessor 具有更高的优先级，主要用来在常规的 BeanFactoryPostProcessor 检测开始之前注册其他 bean 定义。特别是，你可以通过 BeanDefinitionRegistryPostProcessor 来注册一些常规的 BeanFactoryPostProcessor，因为此时所有常规的 BeanFactoryPostProcessor 都还没开始被处理。

AbstractApplicationContext#registerBeanPostProcessors

加载BeanPostProcessor，往容器中注册bean的后置处理器(可以是我们自定义的)，在bean的生命周期中调用bean的后置处理器，来做一些定制化操作

AbstractApplicationContext#initMessageSource

国际化处理，为上下文实例化并注册一个MessageSource

AbstractApplicationContext#initApplicationEventMulticaster

初始化并注册一个事件广播器用来发布事件
从bean工厂中获取或者直接显式的new一个事件多播器赋值给applicatoinContext对象的applicationEventMulticaster属性，事件多播器采用典型的设计模式就是观察者模式，多播器作为的是一个被观察者

AbstractApplicationContext#onRefresh

空方法，由子类实现，使用的是设计模式中的模板方法，是一个钩子方法, Spring Boot 就是从这个方法进行 tomcat 的启动。

AbstractApplicationContext#registerListeners

注册监听器，在所有注册的bean中查找Listener bean，注册到事件多播器中，即向监听器发布事件

AbstractApplicationContext#finishBeanFactoryInitialization

创建所有的非懒加载的单例bean，核心是getBean(),也是bean生命周期的入口
*初始化此上下文的转换服务
*如果beanFactory 之前没有注册嵌入值解析器，则注册默认的嵌入值解析器:主要用于注解属性值的解析
*初始化LoadTimeWeaverAware Bean实例对象
*禁用临时类加载器
*冻结所有bean定义，注册的bean定义不会被修改或进一步后处理，因为马上要创建 Bean 实例对象了
*实例化所有剩余(非加载)单例对象

AbstractApplicationContext#finishRefresh

完成刷新过程,发布刷新事件（Spring cloud eureka 也是从这里启动的）
*清除上下文级别资源缓存
*为该上下文初始化生命周期处理器
*获取生命周期处理器并执行刷新
*发布上下文刷新事件(即广播事件)
